## What this project is

TANIK (Turkish for "witness") is an open-source multi-modal biometric authentication kiosk. It fuses iris recognition (via `worldcoin/open-iris`) with fingerprint matching (via SourceAFIS) behind a FastAPI inference service, fronted by a Next.js client.

The primary goal of this project is **not** novel research. It is to be a credible, honest, well-documented reference system that demonstrates domain competence to biometrics-industry engineers — specifically engineers at Proline (Pasifik Teknoloji), where the author is currently placed as a web developer and wants to move closer to the biometrics work.

Read `README.md` and `ROADMAP.md` before suggesting any non-trivial change.

## The phase-gate discipline — this is the most important rule

This project has a phased roadmap in `ROADMAP.md`. Phases ship before the next phase starts. Each phase has an explicit "definition of done" and an explicit "out of scope" list.

When the author asks for something that belongs to a later phase, or that feels like scope creep:

1. **Do not silently absorb it into the current phase.**
2. Point out which phase it belongs to.
3. Offer to add it to `BACKLOG.md` (create the file if it doesn't exist yet).
4. Ask whether they want to defer it or consciously pull it forward.

This project's biggest failure mode is scope drift, not technical difficulty. The author has a documented pattern of restarting projects when they get too complex. Protecting focus is genuinely more valuable than being helpful by adding features.

Corollary: if a phase has not shipped (deployed, documented, committed, definition-of-done met), do not start work on a later phase, even if the later work seems easier or more fun.

## Technical conventions

### Backend (`apps/inference/`)

- Python 3.11+, FastAPI, Pydantic v2, `opencv-python-headless` (never plain `opencv-python` — it breaks in Docker).
- All CPU-bound biometric work MUST be offloaded via `asyncio.to_thread` or `fastapi.concurrency.run_in_threadpool`. FastAPI's event loop must not block on a Hamming distance calculation or a SourceAFIS match.
- Pydantic models validate every request body; no raw file streams accepted blindly.
- File uploads are validated against magic bytes (real MIME), not the `Content-Type` header.
- No print-debugging into production code. Use `logging` with appropriate levels.
- Raw biometric images exist only in request-scoped memory. They are never written to disk and never persisted. Only extracted templates are stored.

### Frontend (`apps/client/`)

- Next.js App Router, React 18+, TypeScript strict mode, Tailwind, shadcn/ui, Zustand.
- The capture flow is a strict state machine. States are enumerated, transitions are explicit, and the UI must not allow a user to, e.g., submit a verification request before the liveness gate has passed.
- Webcam cleanup is non-negotiable: every `MediaStreamTrack` opened in a `useEffect` must be `.stop()`-ed in the cleanup return. Forgetting this causes hard-to-diagnose memory leaks on kiosk hardware.
- Never stream full-framerate video to the backend. Throttle to discrete frame submissions (2-3 fps for liveness checks).

### Infrastructure

- Multi-stage Dockerfiles for both services. The Next.js builder stage produces a standalone output; the runner image contains only the runtime.
- `docker-compose.yml` uses an internal bridge network. The client reaches the inference service via internal hostname (e.g., `http://inference:8000`), not via the public network.
- All thresholds, weights, CORS origins, and API URLs come from environment variables. Never hardcoded.
- GitHub Actions CI runs backend unit tests on every push.

### Documentation

- `docs/` is not decorative. Documents referenced from the README (threat model, performance, fusion methodology, datasets, API contract, sequence flow) must actually exist and stay in sync with the code.
- When code changes in a way that invalidates a doc claim, update the doc in the same commit.
- Performance numbers (FAR, FRR, latency, throughput) are generated by `tests/evaluation/` scripts and read by `docs/performance.md`. They are never typed by hand into the README. If a commit changes the evaluation pipeline, it must regenerate the numbers.

## Honesty conventions

This project's credibility comes from honest metrics and honest threat modeling. Claude Code must enforce this even when shortcuts feel natural.

- **No invented FAR/FRR numbers.** If a number has not been measured on the test set, it does not appear in the README, the API response, or any doc. "TBD — awaiting evaluation run" is acceptable; a made-up "1 in 1,000,000" is not.
- **No overclaiming on liveness.** The v1 presentation-attack-detection module is a basic defense and must be documented as such. Do not describe it as "military grade" or "state of the art." Document what it detects, what it doesn't, and what real iris-PAD would require (NIR hardware features, specialized models).
- **Accurate privacy claims.** The system does not persist raw images; only templates. Templates are personal data under KVKK/GDPR. "Zero-knowledge" is the wrong phrase because templates *are* stored — use precise language.
- **Proper attribution.** Worldcoin `open-iris` and SourceAFIS are credited in the README and in relevant module docstrings. When borrowing methodology (score normalization, fusion weighting, PAD approach) from published work, cite the paper.

## What NOT to do without explicit confirmation

- Do not add face recognition as a third modality. It's in the deferred list and it changes the product.
- Do not build a mobile PWA or native mobile app. The phone-as-reader concept is a separate product and is explicitly captured in `ROADMAP.md` as a distinct v2 direction.
- Do not introduce new ML dependencies beyond `open-iris` and SourceAFIS in the first three phases. Stay on the shoulders of the existing pipelines.
- Do not refactor the architecture "preemptively." If the current structure works for the current phase, leave it.
- Do not sprinkle `TODO` comments. Either do the thing, or write it down in `BACKLOG.md` with enough context to pick up later.
- Do not generate large amounts of "example" or "sample" code in the main codebase. Examples go in `notebooks/` or `examples/`, not in `src/`.
- Do not add analytics, tracking pixels, or telemetry. This is a biometric system; the privacy posture is load-bearing.

## Communication style (author's preferences)

- Direct and honest. If a request is a bad idea, say so and explain why. Pushback is welcome.
- Short over long. A correct two-sentence answer beats a thorough five-paragraph one.
- Cite concrete sources when making technical claims, especially for biometric research or standards (ISO 19794, NIST IR, ICAO 9303).
- Prefer paraphrasing and one-line references over block quotes from external material.
- When uncertain, say so explicitly instead of guessing.
- Don't be sycophantic. Don't open with "Great question!" The work is the appreciation.
